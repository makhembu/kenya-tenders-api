# Phase 5 — Tournament Round 2: 5 → 3 (Stress Fractures)

## Stress Test Protocol
- **ST1: The Safaricom Problem** (Telco feature kill risk).
- **ST2: The Single Customer Test** (Niche fragmentation).
- **ST3: The Free Alternative Test** (Excel/WhatsApp).
- **ST4: The Data Problem** (External data dependency).
- **ST5: The Churn Cliff** (Long-term value).

---

## Stress Test Results

### 1. BakiPay (Dividend Automation)
- **ST1:** Safaricom's M-Chama focuses on collections, not complex tenure-based dividend calculations. (Score: 0)
- **ST2:** Entirely dependent on Chamas. (Score: -4)
- **ST3:** Chamas currently use manual Excel formulas which are prone to error during dividends. (Score: 0)
- **ST4:** All data is generated by submissons to the API. (Score: 0)
- **ST5:** Value increases as contribution history accumulates. (Score: +2)
- **Adjusted Rank Score:** -2

### 2. StandingWatch (Sports Logic Engine)
- **ST1:** No telco interest in lower-tier local sports standings. (Score: 0)
- **ST2:** Only sports leagues. (Score: -4)
- **ST3:** Heavy competition from local sports bloggers and Excel, but none offer a verified API for real-time widgets. (Score: -2)
- **ST4:** Depends on match data entry which must be accurate to be useful. (Score: -2)
- **ST5:** Historical archives become the "official" record of the league. (Score: 0)
- **Adjusted Rank Score:** -8

### 3. LeagueCenter API (League Management)
- **ST1:** Very low risk of telco involvement in league admin. (Score: 0)
- **ST2:** Only sports branches. (Score: -4)
- **ST3:** Facebook and WhatsApp groups are free and currently "good enough" for most clubs. (Score: -5)
- **ST4:** Fully internal data. (Score: 0)
- **ST5:** Convenience only. (Score: -2)
- **Adjusted Rank Score:** -11

### 4. WelfarePoint (Statement API)
- **ST1:** Safaricom could easily add a "Contribution Statement" button to M-Pesa. Adjacent to "M-Pesa for Business". (Score: -3)
- **ST2:** Chamas only. (Score: -4)
- **ST3:** SMS alerts from M-Pesa already provide a "statement" of sorts. (Score: -4)
- **ST4:** Fully internal usage data. (Score: 0)
- **ST5:** Switching costs are low (can move to a spreadsheet in an afternoon). (Score: -1)
- **Adjusted Rank Score:** -12

### 5. SaccoSync (Consolidated Reports)
- **ST1:** Telcos/Banks target individual consumers, not the complex hierarchy of umbrella community orgs. (Score: 0)
- **ST2:** Umbrella organizations, Welfare Societies, and Multi-branch Saccos. (Score: 0)
- **ST3:** High-end ERPs exist but are too expensive; local alternatives are manual Excel consolidation. (Score: 0)
- **ST4:** Internal hierarchy data. (Score: 0)
- **ST5:** Data lock-in is extremely high once sub-groups are synced. (Score: +2)
- **Adjusted Rank Score:** +2

---

## Elimination Post-Mortems

### ELIMINATED: WelfarePoint
[Post-mortem: The idea failed because its core value proposition—statements—is a feature that Safaricom or any bank can (and eventually will) build into their own business apps for free. The switching cost is negligible, and the competitive pressure from free M-Pesa transaction histories makes it a commodity product with zero defensibility.]

### ELIMINATED: LeagueCenter API
[Post-mortem: The fundamental failure is that "Facebook is good enough." Community sports leagues in Kenya have operated on social media for a decade. While an API is "better," the administrative overhead of making club secretaries use an API instead of just posting a photo of a scorecard to a WhatsApp group creates an adoption friction that kills the product on arrival.]

1. **SaccoSync** (The "Admin" layer for umbrella orgs)
2. **BakiPay** (The "Dividend" layer for Chamas)
3. **StandingWatch** (The "Official History" layer for sports)
